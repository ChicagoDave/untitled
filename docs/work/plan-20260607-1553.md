### Design Decisions (recorded here; ADRs to be written during implementation)

**Bundle-assembly method (ADR-0017 candidate)**: SwiftPM does not natively emit a `.app` bundle with `Info.plist`. Three approaches were considered: (A) a shell script (or Makefile target) that runs `swift build -c release`, creates `Galley.app/Contents/{MacOS,Resources}`, copies the executable, and writes `Info.plist` from a template; (B) wrapping the project in an Xcode project that manages the bundle; (C) using a third-party SwiftPM plugin (e.g. `create-dmg`, `swift-bundler`). Approach A is chosen: it preserves the `swift build`/`swift test` spine (ADR-0011), adds no Xcode artifact to maintain, and keeps the bundle assembly transparent and auditable. The script lives at `scripts/bundle.sh` in the repo root. The `swift run` development path is unchanged. This choice must be recorded as an ADR because it constrains how future packaging (code signing, notarization) is added.

**Open-document routing (ADR-0018 candidate)**: The app uses `WindowGroup` + `@NSApplicationDelegateAdaptor AppDelegate` (not `DocumentGroup`). Two approaches to receive Launch Services open-document events: (A) keep `WindowGroup` and handle `application(_:openFile:)` / `application(_:openURLs:)` in `AppDelegate`, routing into the existing `WorkspaceModel.open(url:)` via the `@MainActor` context; (B) adopt `DocumentGroup`, which would require restructuring the entire app around `FileDocument` or `ReferenceFileDocument`. Approach A is chosen: it is low-risk (the routing function is one method), preserves ADR-0011's separation (no AppKit leaks into `GalleyShell`), and keeps the single-window workspace model intact. The `AppDelegate` already exists for the activation-policy fix; adding one delegate method is additive. This choice must be recorded as an ADR because it determines how all future file-open flows (drag-to-dock, Finder double-click, `open` CLI) are dispatched.

**UTI and package type (complements ADR-0007)**: The `.galley` type is registered as a macOS package — a directory that Finder presents as a single opaque file — via `CFBundleDocumentTypes` and an exported UTI with `UTTypeConformsTo: [com.apple.package, public.composite-content]` and `LSTypeIsPackage: true`. This does NOT change the on-disk layout (`prose.txt` + `sidecar.json` remain plain files inside the directory) and does NOT supersede ADR-0007. A new ADR (0019 candidate) should complement ADR-0007 explicitly, stating: (i) the package boundary is a Finder/OS perception layer only; (ii) `prose.txt` is always independently openable with any text editor; (iii) switching to a zip or binary container is explicitly ruled out.

### Design Decisions (recorded here; ADRs to be written during implementation)

**Unsaved-New buffer policy (ADR-0015 candidate)**: A blank buffer created by Cmd-N has no `fileURL`. Switching away from it does NOT prompt and does NOT write a temp file. The buffer lives in memory only. If the buffer has content (at least one non-empty run) at close time, the user is asked to save or discard. A truly empty blank (zero characters in all runs) is discarded silently. Rationale: this matches every macOS text editor's behavior and avoids surprising temp-file pollution.

**Last-buffer-close behavior (ADR-0015 candidate)**: Closing the last open buffer replaces it with a fresh blank rather than quitting. Quitting (Cmd-Q) is a separate explicit action. Rationale: keeps the workspace alive and consistent with browser-tab conventions; avoids accidental quit from Cmd-W on the only document.

**DocumentModel split (fix (a))**: `DocumentModel` mixes two concerns: (i) AppKit panel presentation (`NSOpenPanel` in `open()`, `NSSavePanel` in `saveAs()`) and (ii) buffer state + bundle load/persist (`load(from:)`, `persist(to:)`, `document`, `fileURL`, `hasContent`). Under fix (a) these are separated: a new `WorkspaceDocument` type in `GalleyShell` owns concern (ii) — it takes a `URL?` and has no AppKit import. The AppKit-side `DocumentModel` in the `Galley` executable becomes a thin panel-runner that runs `NSOpenPanel`/`NSSavePanel` and hands resulting URLs into `WorkspaceDocument`/`WorkspaceModel`. `WorkspaceDocument` is the headlessly-testable unit; `DocumentModel` (panel layer) is not tested headlessly.

**Workspace container abstraction (ADR-0016 candidate)**: `WorkspaceModel` is an `@Observable` class in `GalleyShell` that owns `[WorkspaceDocument]` + `currentIndex: Int`. It is AppKit-free and headlessly testable. `Galley.swift` owns exactly one `WorkspaceModel`; `ContentView` receives it and derives `currentDocument`. The executable's `DocumentModel` (panel layer) wraps or bridges into `WorkspaceModel` for the panel-presentation path. This is a store — Boundary Statement required before editing; OWNER is the `GalleyShell` library (app-layer state), not the `Galley` executable.

**Auto-save on switch**: When `switchTo(index:)` is called, the workspace calls `currentDocument.persist(to: fileURL)` if and only if `currentDocument.fileURL != nil` (i.e., the buffer has been saved at least once). Unsaved buffers are left untouched in memory.

### Design decisions already made (co-designed with user; do not re-litigate)

**Layered template library**: Templates load from three layers, merged at load time with most-specific winning on a name collision (story > user > built-in):
- **Built-in templates** — shipped with the app, available at runtime even in a new/blank buffer with no project on disk. Implemented as in-code `BlockTemplate` values (or bundled resources) in `GalleyShell` — no disk folder required.
- **User templates** — a global cross-project library at `~/Library/Application Support/Galley/templates/`. Plain `.galley-template` files the writer edits in any text editor.
- **Story-specific templates** — the existing per-project `<project>.galley/templates/` folder (ADR-0022, BP1). No change to this layer.
No seeding or copying between layers. Each layer loads independently; the palette merges them. This **amends ADR-0022's "templates live in the per-project `templates/` folder" decision** to add the built-in and user layers — bible and snippets remain project-scoped (they are this novel's data); block templates are the writer's toolkit and are cross-project. ADR-0020 (bible/snippets in the app layer) is unchanged and needs no amendment.

**Authoring / typesetting boundary (ADR-0024, recorded this session)**: Galley captures manuscript content + structure + *intent*; it never paginates, lays out pages, vertically-centers, or renders final image placement. Typesetting is a separate process (in-app / CLI / separate tool — open question) that consumes the `.galley` format. Consequence for this track: structural roles and asset references are stored as **intent**; output layout (the dedication's vertical centering, new-page breaks) is OUT of Galley scope, deferred to the typesetting process.

**Dedication = a section insert, not a template**: A dedication is a short titled section with a paragraph of prose — not a styled single block, not inherently italic, and (in output) vertically centered on its own page. The vertical centering is a typesetter concern (ADR-0024), out of scope here. In Galley a dedication is the same mechanism as prologue/epilogue: a roled `ChapterCut` + fresh prose. Removed from the template layer.

**Prologue / Epilogue / Dedication / Chapter = section inserts via a closed `SectionRole` (not block types, not bare titles)**: A section of prose is structurally a chapter (movable `ChapterCut` overlay, ADR-0005). Because the typesetter must drive page layout from these (ADR-0024), the cut carries a **closed `SectionRole` enum** (`chapter | prologue | epilogue | dedication`) — intent the typesetter consumes — rather than a fragile title string. A section insert places a `ChapterCut` with the chosen role, **anchored to the freshly-seeded prose block** (so the role labels the right section), and positions the caret into that prose. The new work: add `SectionRole` to `ChapterCut` (closed, ADR-0009-style; serialized in the sidecar) and an insert path (a pure `InputEvent.insertSection(role:afterBlockID:)` reducer arm placing the roled cut + seeding a fresh paragraph, or an app-layer composition of `placeChapterCut`/`setChapterCutTitle`/`insertBlock` if it carries no partial-state risk — decide and record in ADR-0026). `ChapterHeading` block types, chapter numbering that skips prologues, and a real TOC remain deferred.

**Unified palette**: Cmd-; lists three kinds of insert, merged in this display order: (1) Built-in blocks (Scene Break — existing); (2) Templates — the 3-layer merged set (Epigraph, Dateline, Block Quote, + user/story); (3) Section inserts — Prologue, Epilogue, Dedication, Chapter (each places a roled `ChapterCut` + new prose block). Built-in set-pieces (verse/epigraph/letter) remain deferred (non-editable, ADR-0023 note).

**Images = deferred future feature (asset reference, not rendering)**: Parked as its own feature. When built: a new closed-vocabulary `figure`/`image` block referencing a package `images/` file with a caption, placed anywhere in the flat block stream (chapter-start or mid-chapter needs no special mechanism — it is just a block's position), shown as a placeholder in the editor; the typesetter places/sizes the real image (ADR-0024). Not in this track.

## LT4: Image/Figure Placeholders

**Added**: 2026-06-07
**Relationship to surrounding phases**: LT1–LT3 are COMPLETE and committed on branch `feat/lt2-lt3-section-inserts-numbering`. This track is independent; it introduces a single new closed `BlockContent` case (`figure`) across three layers — `GalleyCore` vocabulary + codec, `GalleyShell` / `GalleyCore` projections, and the `Galley` executable placeholder UI + palette row. The authoring/typesetting boundary (ADR-0024) is the central constraint: Galley records intent only; the typesetter places and sizes the actual image.
**Bounded contexts touched**: N/A — closed vocabulary extension (ADR-0009) + authoring-layer UI only; typesetting is explicitly out of scope (ADR-0024).
**Key domain language**: figure block, imageRef, caption, placeholder, `images/` package directory, typesetter intent

### Design decisions to resolve early (candidate ADRs — deliberate before any code is written)

**ADR-0027 candidate — figure block shape and serialization**: A `figure` block carries two pieces of intent: the image reference (a filename relative to the package's `images/` directory) and a caption (human-readable text). Two sub-decisions must be recorded before coding:

1. **Block shape**: `BlockContent.figure(imageRef: String, caption: String)` — a minimal, closed struct-like case. Caption is plain text (no inline italic runs) in v1; keeping it flat avoids the `[Run]` deserialization complexity and matches the "closed, minimal" principle (ADR-0009). If italic captions are needed, that is a future amendment, not a v1 design question. `imageRef` is a bare filename (`photo.jpg`), not a URL or path; the package directory is implicit (ADR-0020's pattern — the indexing layer knows the package root).

2. **Serialization strategy**: A `figure` block has no prose text of its own, exactly like `sceneBreak`. Two options: (A) a dedicated prose marker line (e.g. `[figure: photo.jpg]` — a bracketed, unambiguous line that `parseProseBlocks` recognizes as a non-prose block) plus the `imageRef` and caption in the sidecar `BlockDTO`; (B) prose-free, sidecar-only (the block is omitted from prose — like `empty:true` — and the sidecar `BlockDTO` carries a `kind: "figure"` field plus `imageRef`/`caption`). Option A keeps the prose file meaningfully readable by a human or downstream tool (ADR-0007's spirit: "writer-owned, diffable"). Option B requires no new prose parsing but makes the prose invisible for figure blocks. **The planner flags this as the single most consequential design decision for this track; it must be deliberated and recorded in ADR-0027 before `Storage.swift` is touched.** Either option is workable; the plan below assumes Option A as the recommendation (human-readable marker line in prose, metadata in sidecar), but the session must verify this choice.

3. **`images/` package directory**: The directory follows ADR-0020's pattern (per-project package subdirectory). For v1, images are placed there manually by the writer (consistent with `bible/`, `snippets/`, `templates/`). No in-app import/copy affordance in v1. The directory need not be pre-created at open time; it is created on first save if any figure block references it (or lazily on manual placement). `WorkspaceDocument` validates figure refs on open (warn if a referenced file is missing — do not reject the document). The `images/` directory is assets, not an index — no `ImageIndex` struct is needed; `imageRef` validation is a simple `FileManager.fileExists` check.

**ADR-0028 — caption editability and EditorLayout segment model (RESOLVED → Option A)**: A figure block's placeholder in the editor has two pieces of displayed content: (a) the placeholder box (non-editable — icon + imageRef text, rendered as an `NSTextAttachment`, like the delete-prompt chip in LT3e) and (b) the caption. **Decision: Option A — the caption is plain-text editable**, displayed directly below the placeholder box as a normal text segment the writer edits in place, routed to a new `InputEvent.setFigureCaption(blockID:caption:)` reducer arm; it requires an `EditorLayout` caption segment (editable, mapping to the figure block's ID). Option B (caption non-editable, set only at insert) was rejected: it would insert a block the caret cannot enter *at all*, which sits against ADR-0023's "real, editable block" promise (plan-review tension) — Option A gives the inserted figure a real keyboard-reachable surface, like a chapter title. This determines Phase LT4-2 scope (the `setFigureCaption` arm and caption segment are in.)

### Architecture constraints (carry-forward, no re-litigation)

- `GalleyCore` stays UI-free (ADR-0002, ADR-0011). `BlockContent.figure`, its `Storage` codec, `revealProjection` chip, and `displayProjection` token all live in `GalleyCore`. No AppKit anywhere in those files.
- The `Galley` executable's `Attribution.swift` renders the placeholder via `NSTextAttachment` (same technique as the rounded delete-prompt chip in LT3e). No actual image content is loaded or rendered.
- `GalleyShell`'s `WorkspaceDocument` handles the `images/` directory existence check on save and the ref-validation warning on open. No new index struct.
- `BlockContent.figure` is a **closed, deliberate addition** to the vocabulary (ADR-0009). It passes the "justify to reveal" gate: it surfaces as a `[figure: filename]` chip in the reveal pane, which is an addressable object the writer can see and remove. No further figure-related cases are opened by this addition.
- `insertBlock(content: .figure(…), overrides: [], afterBlockID:)` reuses the existing reducer arm (ADR-0023) — no new `InputEvent` case is needed for insertion. A `setFigureCaption(blockID:caption:)` arm may be needed for caption editing (ADR-0028 decision).
- The palette inserts a figure via the existing `insertBlock` path, with the imageRef seeded to an empty string (the writer fills it in by clicking/editing). Mnemonic key: **G** ("Figure" — G is the most natural free letter given F is likely taken by other rows; `assignKeys` will resolve at implementation time).

4. Undo/redo dispatch is shared/workspace-level (already hoisted in UX-1 to `WorkspaceDocument.undo`/`redo`; `performUndo`/`performRedo` in `InputController+Title.swift` must be lifted to a shared owner so the reveal surface can also trigger them).

6. `GalleyCore` stays UI-free; the reveal *view* is AppKit in the `Galley` executable only.
