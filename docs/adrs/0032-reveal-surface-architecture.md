# ADR-0032 — Reveal surface architecture: a second NSTextView fed by a model-annotated projection

## Context

[ADR-0030](0030-reveal-codes-is-a-co-equal-editing-surface.md) decided that Reveal Codes becomes a co-equal TextKit editing surface that writes the *same* `InputEvent`s against the *same* `WorkspaceDocument` as the prose editor, superseding [ADR-0014](0014-reveal-pane-swiftui-flow-layout.md)'s SwiftUI `FlowLayout` of `Text`+chip capsules (`RevealPane.swift`). It left the surface *mechanism* to LT5 planning. Two things must be decided before any reveal code is written:

1. **What kind of view the reveal surface is** — a dedicated `NSTextView` subclass paralleling the prose `InputController`, or a single controller driving two text views.

2. **How a reveal character position maps to a model `(blockID, offset)` and back** — the load-bearing seam. The prose editor solves this with `EditorLayout` (`EditorLayout.swift`): it *re-walks the `Document`* and records, per rendered segment, the `BlockID` and the segment's character range, giving `modelPosition(forCharacterAt:)` / `characterPosition(forBlock:offset:)`. The reveal side has no equivalent. The existing pure projection `Document.revealProjection() -> [RevealToken]` (`Reveal.swift`) is a **flat stream that loses model coordinates**: a `.text(String)` token carries no `BlockID` and no in-block offset, and a single paragraph emits *several* text tokens split by `[i]`/`[/i]`/`[Chapter]` codes. A plain paragraph emits a bare `.text` with no surrounding code to infer a block from. So a reveal caret cannot be mapped to the model from the flat token stream alone.

The prose editor (`InputController`) is an `NSTextView` subclass that renders from the model via `EditorLayout`, intercepts keystrokes, and dispatches `InputEvent`s through `applyEdit(_:)` → `WorkspaceDocument.apply(_:caret:)`. It is the proven pattern; the reveal surface should reuse it rather than invent a second input architecture.

## Decision

1. **The reveal surface is a dedicated `NSTextView` subclass, `RevealController`, mirroring `InputController`.** It owns a `RevealLayout` (the analogue of `EditorLayout`), renders the reveal projection into its text storage on each model change, and — in later phases — intercepts keystrokes and dispatches `InputEvent`s. It is **not** a shared controller driving two views: the two controllers stay decoupled and communicate *only* through `WorkspaceDocument` (the model and the shared caret, [ADR-0033](0033-shared-caret-ownership.md)). This is the [ADR-0004](0004-own-model-is-source-of-truth.md) guarantee — both surfaces are pure projections-of and writers-to the one model, so they cannot drift — applied to the view layer: no view-to-view wiring exists to drift.

2. **A pure, model-annotated reveal projection is introduced; the flat `revealProjection()` is not used by the editing surface.** `RevealLayout` does not consume `[RevealToken]`. Instead a pure function emits `[RevealSegment]`, where each segment is either prose text or a code chip and **every segment carries its model coordinates**: the `BlockID` it belongs to (where one applies) and, for editable prose, the in-block character-offset range it covers. `RevealLayout` renders these segments to an `NSAttributedString` and records a `segments` array exactly as `EditorLayout` does, giving the same `modelPosition`/`characterPosition` contract on the reveal side. The annotated projection is **pure (no AppKit)** and lives in `GalleyShell` so it is headlessly testable (ADR-0011); only the `NSAttributedString` construction and segment-range bookkeeping live in `RevealController`/`RevealLayout` in the `Galley` executable.

3. **The annotated projection is derived by re-walking the `Document`, the same source of truth `revealProjection()` walks** — not by parsing the flat token stream. To avoid drift between the two projections, the shared code-emission decisions (which codes a block emits, their labels, italic-span numbering, override and cut interleaving) stay expressed once. The flat `revealProjection() -> [RevealToken]` in `GalleyCore` is retained unchanged for the pure-value tests and any non-editing consumer; the annotated walk in `GalleyShell` is the editing surface's input. Where practical the annotated walk is built so its code-emission order is asserted (in tests) to match `revealProjection()`'s, so a divergence is caught.

4. **`GalleyCore` stays UI-free** (ADR-0002, ADR-0011). The annotated projection is a `GalleyShell` value type over `GalleyCore` model values; `RevealLayout`/`RevealController` are `Galley`-executable AppKit. No rendering type enters the core.

## Consequences

- **The reveal surface gains the prose editor's caret-mapping contract.** `RevealLayout` exposes `modelPosition(forCharacterAt:)` and `characterPosition(for caret: Caret)`, so the one shared caret ([ADR-0033](0033-shared-caret-ownership.md)) can be rendered into reveal coordinates and a reveal click can be written back to the model — the "one selection rendered twice" requirement of ADR-0030 §Decision 2.
- **Two projections of the same document now exist** (`[RevealToken]` flat and `[RevealSegment]` annotated). This is a deliberate cost: the annotated one carries coordinates the flat one structurally cannot. The drift risk is mitigated by deriving both from the same `Document` walk and asserting their code order agrees in tests. If maintenance proves the two hard to keep in step, a later ADR may fold the flat projection into the annotated one (deriving `[RevealToken]` from `[RevealSegment]`); that is explicitly *not* done now to keep LT5-1 scoped.
- **`RevealItem`/`revealItems(from:)` and the SwiftUI chip views are retired** for the editing surface (the ADR-0014 `FlowLayout` path). They may remain only if a non-editing consumer needs them; otherwise they are removed in LT5-1.
- **The step-over-a-code-as-one-unit behavior has a home.** Because each segment knows whether it is a code chip (non-editable, atomic) or editable prose, `RevealController` can snap the caret off a chip exactly as `InputController.skipPastHeading` snaps it off a chapter heading — the same proven mechanism.
- **`RevealController` is read-only *as to the document model* in LT5-1.** It renders and positions the caret and **does write the shared `currentCaret`** ([ADR-0033](0033-shared-caret-ownership.md)) when its own selection moves — a caret write is not a model edit. What it does not do in LT5-1 is dispatch any model-mutating `InputEvent`; bidirectional editing and the code→event table are [LT5-2] / ADR-0034. "Read-only" here means *no `InputEvent` dispatch*, not *no caret participation*. This keeps the hardest seam (caret ownership + mapping) provable before any reverse-editing risk is added.
- **Boundary:** `RevealLayout`/the annotated projection are projections/read-models. A Boundary Statement is produced before adding them; they own no state — they are pure functions over `Document` plus the shared caret read from `WorkspaceDocument`.
