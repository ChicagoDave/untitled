# ADR-0033 — Shared-caret ownership: one published `currentCaret` on `WorkspaceDocument`

## Context

[ADR-0030](0030-reveal-codes-is-a-co-equal-editing-surface.md) requires **one model-coordinate selection shared by both panes** — "the reveal caret matches the editor" is *one selection rendered twice*, not two carets kept in sync. [ADR-0031](0031-undo-redo-carries-the-caret.md) (implemented this session, commit 5364c63) introduced the `Caret` value type (`app/Sources/GalleyShell/Caret.swift` — a model-coordinate `start`/`end` selection, collapsed when `start == end`) and made the undo timeline store `(Document, Caret?)` per entry.

But the *live* caret today still lives **inside the prose view**: it is the `NSTextView` selection, read on demand via `InputController.caretModelSelection()` (which maps the view's `selectedRange()` through `EditorLayout`). There is no caret value owned outside the view. With a single editing surface that was fine. With two editing surfaces (`InputController` and the new `RevealController`, [ADR-0032](0032-reveal-surface-architecture.md)) it is not: neither view can be the home of the caret, because the other view must read and write the same value, and a view that does not have focus has a stale or meaningless selection.

`ADR-0030`'s Consequences already named the requirement: "the caret must live in model coordinates owned at the `WorkspaceDocument`/workspace level (or a shared controller), not privately inside one `InputController`." And: "Cmd-Z/Cmd-Shift-Z dispatch to the one workspace-level undo timeline regardless of which pane has focus; the restored selection updates *both* panes." Today `performUndo`/`performRedo` live privately in `InputController+Title.swift`. This ADR fixes *where the live caret lives*; the undo hoist is the mechanical consequence.

## Decision

1. **`WorkspaceDocument` owns one published live caret: `var currentCaret: Caret?`.** It is model-coordinate (the `Caret` type from ADR-0031), `@MainActor`, and observed by both surfaces via `@Observable`/`@Bindable`. `WorkspaceDocument` already owns the undo timeline (which stores carets) and is the single shared store both panes bind to; the live caret is per-document editing state it legitimately owns, not per-render view state. A new shared "editing coordinator" object was considered and rejected: it would be a second shared owner alongside the store, and the store already is the one thing both panes hold.

2. **Each surface writes `currentCaret` when its own selection changes, and reads it to reconcile after a model change.** When a view has focus and its selection moves, it maps the view selection to a `Caret` (via its `*Layout.modelPosition`) and writes `currentCaret`. After any model mutation (an edit or an undo/redo), each view reads `currentCaret` and maps it back to its own character coordinates (via its `*Layout.characterPosition(for:)`) to place its selection. A re-entrancy guard (the existing `withoutSelectionSync` pattern in `InputController`) prevents a write-triggered read from looping.

   **The *post-edit* caret is written too, not only the user-moved caret.** Today an edit computes where the caret should land and places it directly (`renderFromModel(caret:)` in `InputController`). That landing position must also be written to `currentCaret` immediately after the edit — so that the value `undo()` later pushes onto the redo stack is the real post-edit caret, not a stale pre-edit one. Without this, redo would restore the wrong caret, violating [ADR-0031](0031-undo-redo-carries-the-caret.md)'s "redo lands the caret after the re-applied change" contract. Concretely: the dispatch path is *write pre-edit `currentCaret` → `apply(event, caret: currentCaret)` → write post-edit `currentCaret` from the computed landing*. Both writes go through the same `currentCaret` cell.

3. **The caret stored in the undo timeline is `currentCaret`.** `WorkspaceDocument.apply(_:caret:)` already accepts the pre-edit caret; the input layer passes `currentCaret`. On undo/redo, the restored `Caret` is assigned back to `currentCaret`, and both surfaces re-read it — so a restore updates both panes (the ADR-0030 requirement). This makes `currentCaret` and the undo-stored caret the same representation, not two.

4. **`performUndo`/`performRedo` are hoisted out of `InputController+Title.swift` to a shared site so either pane triggers them.** The mechanics live as a small shared step (methods on `WorkspaceDocument` that mutate the model and set `currentCaret`, with each view's key handler calling them and then reconciling its selection from `currentCaret`). `Cmd-Z`/`Cmd-Shift-Z` work from whichever pane has focus; the restored selection updates both; focus stays with the active pane.

## Consequences

- **`currentCaret` and the per-snapshot undo caret are the same value type, set in the same places** — no parallel caret state to drift. The live caret is just the "current" cell of the timeline the undo stack records snapshots of.
- **`InputController` stops being the caret's home.** `caretModelSelection()` becomes the *writer* of `currentCaret` (on selection change) rather than the canonical reader; reads for edit/undo come from `currentCaret`. This is a behavioral refactor of the prose editor even before the reveal surface edits — verified by the existing prose smoke checks plus the LT5-1 caret-sync check.
- **A selection-change observer is needed on each view.** `InputController` already has selection-driven logic (`syncTitleEditingToCaret`); writing `currentCaret` hooks into the same settled-selection path. `RevealController` adds the symmetric hook in LT5-1.
- **Re-entrancy is the main hazard.** View A writes `currentCaret` → observation fires → view B reconciles (fine) but view A must not re-reconcile from its own write. The `withoutSelectionSync` suppression already used for re-render is extended to cover caret writes. This is called out as the thing the LT5-1 GUI smoke check must stress (move caret rapidly in one pane; confirm no flicker/loop).
- **Boundary:** `WorkspaceDocument` is a store (rule 8a). A Boundary Statement is produced before adding `currentCaret` — it is per-document editing state the store already-owning-the-timeline owns; it is *not* per-consumer view state, which is exactly why it cannot live in either view (the ADR-0030 shared-caret requirement is the proof).
- **Unblocks bidirectional editing.** With one caret both surfaces read/write, [LT5-2] can dispatch reveal edits and have the prose pane follow with no view-to-view coupling.
