## Completed

### Phase 3 — Editing via pure `InputEvent`/`applyInput` reducer (ADR-0013)

Core additions: `InputEvent.swift` defines the discriminated union of editing operations — `insert`, `splitParagraph`, `mergeParagraph`, `deleteBackward`, `toggleItalic`, `makeSceneBreak`, `toggleSetPiece`, `breakSetPieceLine`. `Input.swift` contains `applyInput(_:to:)`, a pure reducer over `Document` that dispatches each event to the existing block-lifecycle ops (from Phase 3 build step 1). `Typography.swift` implements `smartTypography(for:after:)` — maps straight quotes to curly, `--` to em-dash, and `...` to ellipsis.

Shell additions: `EditorLayout.swift` manages the `NSTextView` caret↔model-index map — converts a `Document` to an `NSAttributedString` via `Attribution`, and translates `NSRange` cursor positions back to `(blockIndex, offset)` pairs for the reducer. `InputController.swift` is an `NSTextView` subclass that overrides `keyDown`, `insertText`, `deleteBackward`, and menu actions to intercept editing intent, build the appropriate `InputEvent`, call `applyInput`, and redraw from the updated model. `DocumentTextView.swift` was refactored to host `InputController` as the editable text view, replacing the prior read-only view.

Two live bugs found and fixed:
1. TextKit 2 blank-repaint after model update: `NSTextView` did not repaint after content-storage swap. Fixed by calling `layoutManager.layoutViewport()` + `needsDisplay = true` after each `performEditingTransaction`.
2. Zero-block document caret target: a freshly created document had no blocks, so there was nowhere to place the caret. Fixed by seeding an empty `.paragraph("")` in `Document.init()` and `DocumentModel.newDocument()`.

Behavioral tests in `InputTests.swift`: 39 tests covering insert, split, merge, deleteBackward, toggleItalic, makeSceneBreak, toggleSetPiece, breakSetPieceLine, and smartTypography edge cases. Test suite graded GREEN before committing.

`TypographyTests.swift`: 8 tests for `smartTypography` — straight-to-curly quote pairs, double-dash → em-dash, ellipsis. GREEN.

## Key Decisions

### 1. ADR-0013 — Editing via pure reducer over intercepted input
`InputController` intercepts `NSTextView` events and translates them to `InputEvent` values; `applyInput` is a pure `(InputEvent, Document) -> Document` reducer. The shell never mutates the model directly — it always goes through the reducer. This keeps the editing contract headlessly testable and separates AppKit mechanics from domain logic.

### 2. ADR-0014 — Reveal pane as SwiftUI flow layout
The reveal pane renders `[RevealToken]` as a wrapped chip stream using a custom SwiftUI `Layout` (no UIKit/AppKit wrapping, no external dependency). Chapter-cut boundaries are overlaid as anchor chips that open the slicing editor inline. This keeps the reveal surface declarative and avoids the NSTextView re-entrancy that a mixed-mode approach would require.
