## Completed

### Storage.swift — parse/serialize with JSON sidecar
`serialize(_ doc: Document) -> (prose: String, sidecar: String)` and `parse(proseText:sidecar:) throws -> Document` plus a `ParseError` enum. Prose uses Fountain-for-prose syntax (`_italic_`, `***` scene break, `:::verse … :::` fences, backslash/underscore and marker-collision escaping). Sidecar is deterministic sorted-key JSON carrying `nextBlockID`, metadata, ordered `{blockID, overrides}`, cuts, and bible — block content never leaks into the sidecar (ADR-0007/0010). Lossless invariant `parse(serialize(doc)) == doc` holds across all block kinds including the `nextBlockID` counter.

### RevealToken.swift — `RevealToken` and `CodeID` enums
`RevealToken` (`.text(String)`, `.code(label:id:)`) and `CodeID` (`.sceneBreak`, `.setPieceOpen/Close`, `.line`, `.chapter`, `.italicOpen/Close`). Pure values, no rendering types (ADR-0006).

### Reveal.swift — `Document.revealProjection() -> [RevealToken]`
Flat token stream: `[SceneBreak]`, `[Verse]/[line]/[/Verse]` (and Epigraph/Letter), `[Chapter]` chips at cuts, and `[i]`/`[/i]` around explicit italic runs. Paragraph rendering uses an offset-merged single pass so a chapter cut inside an italic span splits cleanly.

### StorageTests.swift — 16 tests
Full round-trip across every block kind, empty document, `nextBlockID` preservation, escaping (underscore, marker collision), blank set-piece lines, no-sidecar import, and 6 rejection tests (`ParseError.unterminatedFence`, `.unknownKind`, `.blockCountMismatch`, `.unknownCutAnchor`, `.unknownOverrideToken`, `.malformedJSON`).

### RevealTests.swift — 13 tests
Paragraph italic chips, plain paragraph, multiple italic spans, scene break, verse/epigraph/letter labels, blank verse line, boundary chapter cut, mid-paragraph chapter cut, chapter-cut-inside-italic interleaving, chapter cut on set-piece, set-piece derived-vs-explicit italic, empty document.

## Key Decisions

### 1. Explicit inline italic IS a reveal code (ADR-0009 refinement)
Phase 4 initially scoped italic out of `revealProjection`, citing the §7 verse example as justification. On review the justification was flawed — verse italic is derived from `SetPieceKind`, not a `Run.italic` mark, so that example proved nothing about explicit marks. Decision reversed: `Run.italic` spans produce `[i]`/`[/i]` chips in the token stream per ADR-0009's principle that "every code is addressable in reveal"; a set-piece's derived italic produces no chips. Suggest a one-line addition to ADR-0009 or ADR-0006 to record this explicit/derived distinction.

Prior sessions ran on Ubuntu (Swift 6.2, userspace PATH workaround). This session ran on the Mac mini (macOS, Apple Swift 6.3). Build/test entry point is `swift test` in `core/` with no PATH workaround needed. No API differences encountered.
